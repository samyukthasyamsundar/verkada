01 The Scenario

When every device shares the same network, one breach can compromise everything.

Most physical security breaches don't start at the camera — they start at the laptop two desks away. A phishing email, a weak password, a compromised device on the same network. And once an attacker is in, the camera feeds, the footage archives, the admin credentials — it's all reachable.

→Employee laptop compromised via phishing
→Attacker moves laterally across shared LAN
→Camera system, footage, admin access all reachable
✕One entry point. Full access. No containment.
The industry reality

"68% of breaches involve a non-malicious human element — a shared network being the most common attack surface for physical security systems."

The answer isn't better firewall rules. It's separating the network by design.

02 The Solution

Six layers of protection. One secure environment.

01 VPC

Dedicated isolated network

Helix Labs gets their own private slice of AWS. No other tenant can see in or route to it.

02 Subnet Segmentation

Cameras off the corporate LAN

Camera and admin traffic live in separate subnets — a breach in one cannot reach the other.

03 Bastion Host

One door in. Fully logged.

All admin access tunnels through a single IP-locked entry point. Every session is timestamped.

04 Security Groups

Nothing in. Port 443 out only.

Camera servers can call out to storage, but cannot receive any inbound connection from the internet.

05 S3 + IAM

No credentials on disk. Ever.

Temporary role-based access only. CloudTrail + VPC Flow Logs create a complete, searchable audit trail.

06 Terraform (IaC)

Scales without increasing headcount

One parameterized template. Add cameras, offices, or clients — infrastructure scales automatically with no manual config.

Why this matters

An IT admin accidentally leaves a port open on a misconfigured firewall. It happens. But you cannot accidentally add a missing route table entry. The private subnet has no path to the internet — not because of a rule that could be changed, but because the route simply does not exist.

02 The Solution

Zero inbound. Port 443 egress only. Deny-all default posture.

  • 01Security group has no inbound rules from 0.0.0.0/0 — no public internet access whatsoever
  • 02Ingress restricted to SSH (port 22) from the bastion security group ID — scoped to a group, not an IP range
  • 03Egress limited to port 443 HTTPS only — mirrors the outbound-only communication model of Verkada cameras
  • 04AWS security groups are implicit deny-all by default. Every permission is explicitly defined.
Why it matters

Most ransomware and lateral movement attacks rely on initiating a connection into a system. If no inbound connection is possible — not blocked, but architecturally absent — that entire attack surface disappears. The camera server calls out to storage. Nothing can call in.

02 The Solution

Least-privilege IAM. No static credentials. Ever.

  • 01S3 bucket with Block Public Access — no ACL overrides, no public URLs, ever
  • 02EC2 instance profile — temporary credentials via metadata service, no keys on disk
  • 03Policy scoped to one bucket ARN. Wildcards explicitly denied.
s3:PutObject s3:GetObject s3:ListBucket
Three actions. One bucket. Nothing else.
Why it matters

Credential theft is one of the most common cloud attack vectors. With static access keys on disk, a stolen credential gives an attacker open-ended access. Here, credentials auto-expire, can't be exported, and only work for three specific operations on one specific bucket — limiting blast radius to near zero even in a worst-case breach.
