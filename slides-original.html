01 The Scenario

Meet Helix Labs

Scenario

A 200-person biotech company. Two offices in San Francisco. Verkada cameras at every entrance, lab, and parking structure. Their IT team needs centralized remote management and secure cloud storage for all footage.

Functional Requirements

What the system must do

  • Isolate camera systems from corporate LAN — no shared broadcast domain
  • Provide a single authenticated, auditable remote access point for IT admins
  • Store encrypted footage in private cloud — inaccessible from the public internet
  • Enable centralized management of Verkada cameras across both office locations
  • Deploy and tear down the full environment in under two minutes via IaC
Non-functional Requirements

How the system must perform

  • Cost efficiency — pay-as-you-go cloud spend; no dedicated hardware CapEx
  • Ease of maintenance — version-controlled IaC with zero configuration drift
  • Network isolation — enforced by architecture, not configuration alone
  • Auditability & reporting — all access logged; compliant with SOC 2, ISO 27001, HIPAA
  • Multi-client scalability — architecture must template across future Helix Labs deployments

02 The Solution

Network segmentation by architecture

  • 01A /16 VPC gives Helix Labs a dedicated address space — fully isolated from every other AWS tenant by default
  • 02Public subnet (/24) routes through an Internet Gateway for admin access. Private subnet has no IGW route entry.
  • 03The private subnet is unreachable by architecture — no route table entry means no path exists, not just a firewall rule blocking one
  • 04All ingress to private hosts is SSH-proxied through the bastion host in the public subnet via agent forwarding
Why this matters

Firewall rules can be misconfigured. Route tables cannot route to an address that has no entry. The private subnet is unreachable by design, not policy.

02 The Solution

Zero inbound. Port 443 egress only. Deny-all default posture.

  • 01Security group has no inbound rules from 0.0.0.0/0 — no public internet access whatsoever
  • 02Ingress restricted to SSH (port 22) from the bastion security group ID — scoped to a group, not an IP range
  • 03Egress limited to port 443 HTTPS only — mirrors the outbound-only communication model of Verkada cameras
  • 04AWS security groups are implicit deny-all by default. Every permission is explicitly defined.
Think of it this way

The camera server can make a phone call, but it cannot receive one. Even if someone knew exactly where it was, they could not knock on the door.

02 The Solution

Least-privilege IAM. No static credentials. Ever.

  • 01S3 bucket with Block Public Access — no ACL overrides, no public URLs, ever
  • 02EC2 instance profile — temporary credentials via metadata service, no keys on disk
  • 03Policy scoped to one bucket ARN. Wildcards explicitly denied.
s3:PutObject s3:GetObject s3:ListBucket
Three actions. One bucket. Nothing else.
Think of it this way

A key that opens one door and lets the server do exactly three things once inside — even if copied, it's useless everywhere else.
